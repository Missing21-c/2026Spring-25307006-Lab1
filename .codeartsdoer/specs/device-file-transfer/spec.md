# 需求规格文档

## 文档信息
- **功能名称**: 多设备音乐文件互传功能
- **创建日期**: 2025-06-14
- **版本**: v1.0
- **状态**: 草稿

## 1. 概述

### 1.1 功能描述
实现多设备间的音乐文件互传功能，允许用户在两台或多台设备上登录同一应用后，通过该应用将设备上的音乐文件（如MP3文件）传输到其他设备。该功能基于现有的AirDrop模块进行增强，提供完整的设备发现、文件选择、传输管理和接收处理能力。

### 1.2 业务背景
在多设备使用场景下，用户经常需要在不同设备间共享音乐文件。例如：
- 用户在手机A上下载了新的音乐，希望在手机B上也能播放
- 用户在平板上整理了音乐库，希望同步到手机上
- 用户希望与家人或朋友分享本地音乐文件

当前应用已具备基础的设备发现和传输能力，但缺少完整的文件选择、传输状态管理和接收端处理功能。

### 1.3 范围界定

#### 包含范围
- 设备发现与在线状态显示
- 本地音乐文件选择器（支持MP3等音频格式）
- 文件传输发起与进度显示
- 文件接收与自动保存
- 传输历史记录查看
- 多文件批量传输支持
- 传输中断与重试机制

#### 排除范围
- 云端存储同步功能
- 跨网络（非局域网）传输
- 音乐版权验证
- 文件格式转换
- 账号系统与权限管理（依赖现有登录功能）

## 2. 功能需求

### 2.1 设备发现与连接

#### 需求 ID: FR-001
**需求描述**: 当用户打开设备传输功能时，系统应当自动扫描并显示附近已登录同一账号的在线设备列表。

**验收标准**:
```
给定(Given): 用户已登录应用且打开设备传输功能
当(When): 系统完成设备扫描
那么(Then): 显示附近所有在线设备列表，包含设备名称、设备类型、在线状态
```

**优先级**: 高

---

#### 需求 ID: FR-002
**需求描述**: 当设备列表为空时，系统应当显示"未发现附近设备"的提示信息，并提供刷新按钮。

**验收标准**:
```
给定(Given): 用户打开设备传输功能
当(When): 附近没有在线设备
那么(Then): 显示"未发现附近设备"提示，并提供手动刷新按钮
```

**优先级**: 中

---

#### 需求 ID: FR-003
**需求描述**: 当用户点击刷新按钮时，系统应当重新扫描附近设备并更新设备列表。

**验收标准**:
```
给定(Given): 设备列表显示界面
当(When): 用户点击刷新按钮
那么(Then): 显示加载状态，重新扫描设备，更新列表显示
```

**优先级**: 中

---

### 2.2 文件选择

#### 需求 ID: FR-004
**需求描述**: 当用户选择发送文件时，系统应当打开文件选择器，支持浏览设备本地存储中的音频文件。

**验收标准**:
```
给定(Given): 用户已选择目标设备
当(When): 用户点击"选择文件"按钮
那么(Then): 打开文件选择器，默认显示音频文件目录，支持筛选MP3、WAV、FLAC等格式
```

**优先级**: 高

---

#### 需求 ID: FR-005
**需求描述**: 当用户在文件选择器中选择文件时，系统应当显示文件的基本信息（文件名、大小、格式）。

**验收标准**:
```
给定(Given): 文件选择器已打开
当(When): 用户选中一个或多个音频文件
那么(Then): 显示每个文件的名称、大小、格式信息，并显示已选文件数量
```

**优先级**: 高

---

#### 需求 ID: FR-006
**需求描述**: 当用户选择多个文件时，系统应当支持批量选择并显示总文件数和总大小。

**验收标准**:
```
给定(Given): 文件选择器已打开
当(When): 用户选中多个文件
那么(Then): 显示已选文件数量、总大小，并提供全选和取消全选功能
```

**优先级**: 中

---

### 2.3 文件传输

#### 需求 ID: FR-007
**需求描述**: 当用户确认发送文件时，系统应当建立与目标设备的连接并开始传输文件。

**验收标准**:
```
给定(Given): 用户已选择目标设备和待传输文件
当(When): 用户点击"发送"按钮
那么(Then): 建立设备连接，开始传输文件，显示传输进度条
```

**优先级**: 高

---

#### 需求 ID: FR-008
**需求描述**: 在文件传输过程中，系统应当实时显示传输进度（已传输大小、总大小、百分比、传输速度）。

**验收标准**:
```
给定(Given): 文件正在传输中
当(When): 传输进度更新
那么(Then): 实时显示已传输字节数、总字节数、完成百分比、当前传输速度
```

**优先级**: 高

---

#### 需求 ID: FR-009
**需求描述**: 当文件传输完成时，系统应当显示成功提示并更新传输历史记录。

**验收标准**:
```
给定(Given): 文件传输进行中
当(When): 传输完成
那么(Then): 显示"传输成功"提示，记录传输历史（文件名、时间、目标设备），自动关闭传输界面或提供关闭选项
```

**优先级**: 高

---

#### 需求 ID: FR-010
**需求描述**: 当文件传输失败时，系统应当显示失败原因并提供重试选项。

**验收标准**:
```
给定(Given): 文件传输进行中
当(When): 传输失败（网络中断、设备离线等）
那么(Then): 显示失败原因，提供"重试"和"取消"按钮
```

**优先级**: 高

---

#### 需求 ID: FR-011
**需求描述**: 当用户取消传输时，系统应当停止当前传输并清理临时文件。

**验收标准**:
```
给定(Given): 文件正在传输中
当(When): 用户点击"取消"按钮
那么(Then): 立即停止传输，清理未完成的临时文件，恢复到传输前状态
```

**优先级**: 中

---

### 2.4 文件接收

#### 需求 ID: FR-012
**需求描述**: 当收到其他设备发送的文件传输请求时，系统应当显示接收确认对话框，包含发送方设备信息和文件信息。

**验收标准**:
```
给定(Given): 应用在后台运行或前台运行
当(When): 收到文件传输请求
那么(Then): 显示接收确认对话框，包含发送设备名称、文件名、文件大小，提供"接收"和"拒绝"按钮
```

**优先级**: 高

---

#### 需求 ID: FR-013
**需求描述**: 当用户确认接收文件时，系统应当自动保存文件到指定目录并显示接收进度。

**验收标准**:
```
给定(Given): 用户收到文件传输请求
当(When): 用户点击"接收"按钮
那么(Then): 开始接收文件，显示接收进度，完成后自动保存到应用音乐目录
```

**优先级**: 高

---

#### 需求 ID: FR-014
**需求描述**: 当文件接收完成时，系统应当显示成功通知并提供"打开"和"播放"选项。

**验收标准**:
```
给定(Given): 文件正在接收中
当(When): 接收完成
那么(Then): 显示成功通知，提供"打开文件位置"和"立即播放"按钮
```

**优先级**: 中

---

#### 需求 ID: FR-015
**需求描述**: 当用户拒绝接收文件时，系统应当向发送方发送拒绝通知。

**验收标准**:
```
给定(Given): 用户收到文件传输请求
当(When): 用户点击"拒绝"按钮
那么(Then): 关闭确认对话框，向发送方发送拒绝通知，发送方显示"对方拒绝接收"提示
```

**优先级**: 中

---

### 2.5 传输历史

#### 需求 ID: FR-016
**需求描述**: 当用户查看传输历史时，系统应当显示所有传输记录，包含文件名、时间、对方设备、传输方向（发送/接收）、状态。

**验收标准**:
```
给定(Given): 用户打开传输历史界面
当(When): 界面加载完成
那么(Then): 按时间倒序显示所有传输记录，每条记录显示文件名、时间、设备名、方向、状态图标
```

**优先级**: 中

---

#### 需求 ID: FR-017
**需求描述**: 当用户点击传输历史记录时，系统应当显示该传输的详细信息。

**验收标准**:
```
给定(Given): 传输历史列表已显示
当(When): 用户点击某条记录
那么(Then): 显示详细信息：文件名、文件大小、传输时间、耗时、对方设备、文件保存路径
```

**优先级**: 低

---

#### 需求 ID: FR-018
**需求描述**: 系统应当支持清空传输历史记录功能。

**验收标准**:
```
给定(Given): 传输历史界面
当(When): 用户点击"清空历史"按钮并确认
那么(Then): 删除所有历史记录，显示"历史已清空"提示
```

**优先级**: 低

---

#### 需求 ID: FR-019

## 3. 非功能需求

### 3.1 性能需求
- 设备发现时间应不超过5秒
- 文件传输速度应达到局域网传输的合理速度（至少2MB/s）
- 单个文件大小支持上限为500MB
- 批量传输文件数量上限为50个
- 界面响应时间不超过200毫秒

### 3.2 安全性需求
- 仅允许已登录用户使用传输功能
- 仅显示同一账号下的在线设备
- 传输完成后应验证文件完整性
- 接收的文件应保存到应用私有目录，避免权限问题

### 3.3 可用性需求
- 提供清晰的操作引导和状态提示
- 支持中文界面
- 错误提示应包含具体原因和解决建议
- 支持横竖屏自适应显示

### 3.4 兼容性需求
- 支持HarmonyOS 5.1.0及以上版本
- 支持直板机、折叠屏、平板等设备形态
- 支持MP3、WAV、FLAC、AAC等常见音频格式

## 4. 约束条件

### 4.1 技术约束
- 基于HarmonyOS分布式设备管理API实现设备发现
- 使用现有的AirDrop模块作为传输基础
- 文件传输使用Socket或分布式文件系统
- 遵循HarmonyOS应用开发规范

### 4.2 业务约束
- 仅支持局域网内设备传输
- 不支持跨账号传输
- 传输的文件仅限音频文件
- 需要用户授权存储权限

### 4.3 设计约束
- UI风格应与现有应用保持一致
- 使用应用现有的颜色主题和组件风格
- 图标使用系统资源或应用已有资源

## 5. 数据需求

### 5.1 输入数据
- 用户选择的本地音频文件路径
- 目标设备ID
- 用户操作指令（发送、接收、取消、重试等）

### 5.2 输出数据
- 传输进度信息（百分比、速度、已传输大小）
- 传输结果（成功/失败）
- 传输历史记录

### 5.3 数据约束
- 文件路径必须为有效的本地路径
- 设备ID必须为有效的在线设备
- 文件大小不超过500MB
- 文件格式必须在支持的音频格式列表中

## 6. 接口需求

### 6.1 用户接口
- 设备列表界面：显示在线设备，支持刷新
- 文件选择界面：支持浏览、选择、多选音频文件
- 传输进度界面：显示传输进度、速度、状态
- 接收确认对话框：显示发送方和文件信息
- 传输历史界面：显示历史记录列表

### 6.2 系统接口
- 设备管理接口：发现设备、获取设备状态
- 文件系统接口：读取文件、保存文件、获取文件信息
- 通知接口：显示进度通知、结果通知
- 权限接口：请求存储权限、网络权限

## 7. 验收测试场景

### 7.1 正常场景
1. **单文件传输**：用户A选择一个MP3文件发送给用户B的设备，传输成功
2. **多文件批量传输**：用户A选择10个音乐文件发送给用户B，全部传输成功
3. **设备刷新**：用户打开设备列表，手动刷新后发现新上线的设备
4. **查看历史**：用户查看传输历史，显示所有历史记录
5. **接收文件**：用户B收到用户A的传输请求，确认接收，文件保存成功

### 7.2 异常场景
1. **设备离线**：传输过程中目标设备离线，显示失败提示并提供重试
2. **网络中断**：传输过程中网络断开，显示失败提示
3. **文件不存在**：选择的文件已被删除，显示错误提示
4. **存储空间不足**：接收设备空间不足，显示错误提示
5. **权限未授权**：用户未授权存储权限，引导用户授权

### 7.3 边界场景
1. **大文件传输**：传输接近500MB的文件，传输成功
2. **空文件列表**：本地没有音频文件，文件选择器显示空状态
3. **无在线设备**：附近没有在线设备，显示空状态提示
4. **取消传输**：传输进度50%时用户取消，停止传输并清理临时文件

## 8. 依赖关系

### 8.1 前置依赖
- 用户已登录应用账号
- 应用已获得存储权限
- 设备已连接网络（Wi-Fi或移动网络）
- 现有的AirDrop模块功能可用

### 8.2 后置依赖
- 传输的音乐文件可在音乐列表中播放
- 传输历史可被查询和清理

## 9. 术语定义

| 术语 | 定义 |
|------|------|
| 设备发现 | 通过分布式设备管理API扫描附近在线设备的过程 |

| 局域网传输 | 在同一局域网内的设备间进行数据传输 |
| 传输历史 | 记录所有文件传输操作的日志信息 |
