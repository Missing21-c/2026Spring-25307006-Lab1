# 需求规格文档

## 文档信息
- **功能名称**: 底部UI优化 (bottom-ui-optimization)
- **创建日期**: 2026-05-26
- **版本**: v1.0
- **状态**: 草稿

## 1. 概述

### 1.1 功能描述
优化音乐应用首页底部UI显示问题，修复底部区域显示不完整、图标缺失等视觉缺陷，并改进卡片交互方式，提升用户体验。

### 1.2 业务背景
当前音乐应用首页存在以下问题：
1. 底部功能区域显示不完整，仅显示深紫色条状区域，内容被严重裁剪
2. 底部交互图标显示异常，仅显示模糊的粉紫色方块，无法正常识别功能
3. 卡片区域的"播放音乐"和"观看直播"按钮位置不合理，用户期望更简洁的交互方式

这些问题严重影响用户对应用功能的识别和使用，需要进行优化修复。

### 1.3 范围界定

#### 包含范围
- 修复底部区域UI显示不完整问题
- 修复底部功能图标显示异常问题
- 移除卡片右下角的功能按钮
- 实现点击整个卡片区域触发相应功能
- 确保UI风格与整体应用保持一致

#### 排除范围
- 不涉及卡片背景图片的修改
- 不涉及页面导航逻辑的修改
- 不涉及数据模型的修改
- 不涉及其他页面的UI调整

## 2. 功能需求

### 2.1 底部区域显示修复

#### 需求 ID: FR-001
**需求描述**: 当底部区域渲染时，系统应当完整显示底部功能栏的所有内容，包括图标和文字标签。

**验收标准**:
```
给定(Given): 用户在应用首页
当(When): 页面加载完成
那么(Then): 底部区域完整显示，高度为72vp，背景色为#3a3a5a，所有功能图标和文字清晰可见
```

**优先级**: 高

---

#### 需求 ID: FR-002
**需求描述**: 在底部区域布局时，系统应当确保功能项水平均匀分布，每个功能项包含图标和文字两部分。

**验收标准**:
```
给定(Given): 底部区域已渲染
当(When): 显示功能项时
那么(Then): 每个功能项包含一个28x28vp的图标和一个13fp的文字标签，图标和文字垂直居中对齐
```

**优先级**: 高

---

### 2.2 底部图标显示修复

#### 需求 ID: FR-003
**需求描述**: 当底部功能项渲染时，系统应当显示清晰可识别的功能图标，而非模糊的色块。

**验收标准**:
```
给定(Given): 底部功能项已创建
当(When): 渲染图标组件
那么(Then): 图标清晰显示，分辨率为原始资源分辨率，无模糊或失真现象
```

**优先级**: 高

---

#### 需求 ID: FR-004
**需求描述**: 系统应当为每个底部功能项配置正确的图标资源，音乐功能使用音乐图标，直播功能使用直播图标。

**验收标准**:
```
给定(Given): 底部功能项对应音乐列表功能
当(When): 渲染图标
那么(Then): 显示音乐图标(ic_music_icon)
```

```
给定(Given): 底部功能项对应直播功能
当(When): 渲染图标
那么(Then): 显示直播图标(ic_live)
```

**优先级**: 高

---

### 2.3 卡片交互优化

#### 需求 ID: FR-005
**需求描述**: 当卡片组件渲染时，系统应当移除右下角的"播放音乐"和"观看直播"按钮。

**验收标准**:
```
给定(Given): 首页卡片组件已创建
当(When): 渲染卡片内容
那么(Then): 卡片内不包含任何按钮组件，仅显示标题、描述和背景图片
```

**优先级**: 高

---

#### 需求 ID: FR-006
**需求描述**: 当用户点击卡片区域任意位置时，系统应当触发相应的功能导航。

**验收标准**:
```
给定(Given): 用户在首页卡片区域
当(When): 用户点击卡片任意位置
那么(Then): 系统导航到对应的功能页面（音乐列表或直播页面）
```

**优先级**: 高

---

#### 需求 ID: FR-007
**需求描述**: 当用户点击卡片时，系统应当提供视觉反馈效果，表明点击操作已被识别。

**验收标准**:
```
给定(Given): 用户在首页卡片区域
当(When): 用户按下卡片
那么(Then): 卡片显示按下状态效果（如透明度变化或缩放效果）
```

**优先级**: 中

---

### 2.4 UI一致性保障

#### 需求 ID: FR-008
**需求描述**: 系统应当保持优化后的UI风格与整体应用风格一致。

**验收标准**:
```
给定(Given): 底部区域和卡片组件已优化
当(When): 用户查看页面
那么(Then): 底部区域和卡片的颜色、字体、间距与页面其他元素保持一致的设计风格
```

**优先级**: 中

---

## 3. 非功能需求

### 3.1 性能需求
- 底部区域渲染时间应小于100ms
- 卡片点击响应时间应小于50ms
- UI优化不应增加页面加载时间

### 3.2 可用性需求
- 底部功能图标应当清晰可识别，符合用户认知习惯
- 卡片点击区域应当足够大，便于用户操作（最小点击区域不小于48x48vp）
- 交互反馈应当及时且明显

### 3.3 兼容性需求
- 支持不同屏幕尺寸的适配
- 支持横竖屏切换时的UI自适应
- 保持与现有响应式布局系统的兼容

### 3.4 可维护性需求
- 图标资源应当使用资源引用方式，便于统一管理
- 样式参数应当使用资源文件定义，便于主题切换

## 4. 约束条件

### 4.1 技术约束
- 开发语言：ArkTS (HarmonyOS)
- UI框架：ArkUI声明式开发范式
- 必须使用现有的资源系统($r语法)引用图标和样式
- 必须保持与现有Navigation系统的兼容

### 4.2 业务约束
- 底部功能项数量应与现有数据源保持一致（排除评论功能）
- 卡片点击行为应与原有按钮点击行为一致
- 不得改变页面导航逻辑

### 4.3 设计约束
- 底部区域高度固定为72vp
- 底部背景色使用#3a3a5a
- 图标尺寸使用28x28vp
- 文字大小使用13fp
- 保持与应用整体设计语言的一致性

## 5. 数据需求

### 5.1 输入数据
- IndexItem数组：包含功能项的标题、描述、图标、URL等信息
- 当前数据源包含两个功能项：音乐列表和直播

### 5.2 输出数据
- 无新增输出数据需求
- 保持现有的页面导航行为

### 5.3 数据约束
- IndexItem数据结构不得修改
- 功能项过滤逻辑保持不变（排除评论功能）

## 6. 接口需求

### 6.1 用户接口
- 底部功能栏：水平排列的功能项，每项包含图标和文字
- 卡片组件：可点击的区域，包含标题、描述和背景图片

### 6.2 系统接口
- 保持现有的Navigation路由接口
- 保持现有的IndexViewModel数据接口

## 7. 验收测试场景

### 7.1 正常场景
1. 用户打开应用首页，底部区域完整显示两个功能项（音乐、直播）
2. 底部功能项图标清晰可识别，文字标签正确显示
3. 用户点击卡片区域，成功导航到对应功能页面
4. 用户点击底部功能项，成功导航到对应功能页面

### 7.2 异常场景
1. 图标资源加载失败时，显示默认占位图标
2. 数据源为空时，底部区域不显示或显示空状态

### 7.3 边界场景
1. 小屏幕设备上底部区域仍能完整显示
2. 长文字标签能够正确截断或换行
3. 快速连续点击卡片不会导致多次导航

## 9. 术语定义
- **ArkTS**: HarmonyOS的应用开发语言，基于TypeScript扩展
- **ArkUI**: HarmonyOS的声明式UI开发框架
- **vp**: 虚拟像素，密度无关的长度单位
- **fp**: 字体像素，用于字体大小设置
- **Navigation**: HarmonyOS的导航容器组件
