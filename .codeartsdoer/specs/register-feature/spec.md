5# 需求规格文档

## 文档信息
- **项目名称**: MusicHome音乐应用
- **功能模块**: 用户注册功能
- **版本号**: v1.0
- **创建日期**: 2026-05-19
- **最后更新**: 2026-05-19

## 1. 概述

### 1.1 功能简介
为MusicHome HarmonyOS音乐应用添加用户注册功能，允许新用户创建账号并完成身份验证，注册成功后可使用账号登录系统。

### 1.2 业务背景
当前应用已实现登录功能，但缺少用户注册入口。新用户无法自主创建账号，限制了用户增长和应用的可用性。添加注册功能可以完善用户认证体系，提升用户体验。

### 1.3 范围界定
**包含范围**:
- 用户注册页面UI设计与实现
- 注册信息输入验证（用户名、密码、确认密码）
- 注册请求提交与响应处理
- 注册成功后自动登录功能
- 注册失败错误提示
- 与现有登录功能的集成

**不包含范围**:
- 第三方账号注册（如微信、QQ等）
- 手机号或邮箱验证码注册
- 密码找回功能
- 用户协议与隐私政策页面

## 2. 用户角色

### 2.1 角色定义
| 角色名称 | 角色描述 | 权限说明 |
|---------|---------|---------|
| 未注册用户 | 没有系统账号的访客 | 可以注册新账号、查看公开内容 |
| 已注册用户 | 已完成注册的用户 | 可以登录系统、使用音乐功能 |

## 3. 功能需求

### 3.1 注册页面展示

#### REQ-REG-001: 注册页面入口
**类型**: 功能需求
**优先级**: 高
**描述**: 系统应在登录页面提供注册入口，允许用户进入注册页面。

**验收标准**:
- 当用户在登录页面点击"注册账号"链接时，系统应导航到注册页面
- 当注册页面加载完成时，系统应显示注册表单界面
- 当用户点击返回按钮时，系统应返回到登录页面

**前置条件**:
- 用户未登录
- 登录页面已加载

**后置条件**:
- 注册页面已显示

#### REQ-REG-002: 注册表单展示
**类型**: 功能需求
**优先级**: 高
**描述**: 系统应展示包含用户名、密码、确认密码输入框的注册表单。

**验收标准**:
- 当注册页面加载时，系统应显示用户名输入框
- 当注册页面加载时，系统应显示密码输入框
- 当注册页面加载时，系统应显示确认密码输入框
- 当注册页面加载时，系统应显示"注册"按钮
- 当注册页面加载时，系统应显示返回登录页面的链接

**前置条件**:
- 注册页面已加载

**后置条件**:
- 注册表单已完整展示

### 3.2 输入验证

#### REQ-REG-003: 用户名格式验证
**类型**: 功能需求
**优先级**: 高
**描述**: 系统应验证用户名格式符合规则要求。

**验收标准**:
- 当用户输入的用户名为空时，系统应提示"请输入用户名"
- 当用户输入的用户名长度小于3位时，系统应提示"用户名长度不能少于3位"
- 当用户输入的用户名长度大于20位时，系统应提示"用户名长度不能超过20位"
- 当用户输入的用户名包含非字母数字下划线字符时，系统应提示"用户名只能包含字母、数字和下划线"
- 当用户输入的用户名格式正确时，系统应不显示错误提示

**前置条件**:
- 用户正在填写注册表单

**后置条件**:
- 用户名验证结果已显示

#### REQ-REG-004: 密码格式验证
**类型**: 功能需求
**优先级**: 高
**描述**: 系统应验证密码格式符合安全要求。

**验收标准**:
- 当用户输入的密码为空时，系统应提示"请输入密码"
- 当用户输入的密码长度小于6位时，系统应提示"密码长度不能少于6位"
- 当用户输入的密码长度大于20位时，系统应提示"密码长度不能超过20位"
- 当用户输入的密码格式正确时，系统应不显示错误提示

**前置条件**:
- 用户正在填写注册表单

**后置条件**:
- 密码验证结果已显示

#### REQ-REG-005: 确认密码验证
**类型**: 功能需求
**优先级**: 高
**描述**: 系统应验证确认密码与密码一致。

**验收标准**:
- 当用户输入的确认密码为空时，系统应提示"请再次输入密码"
- 当用户输入的确认密码与密码不一致时，系统应提示"两次输入的密码不一致"
- 当用户输入的确认密码与密码一致时，系统应不显示错误提示

**前置条件**:
- 用户已输入密码
- 用户正在输入确认密码

**后置条件**:
- 确认密码验证结果已显示

### 3.3 注册请求处理

#### REQ-REG-006: 注册请求提交
**类型**: 功能需求
**优先级**: 高
**描述**: 当用户点击注册按钮时，系统应提交注册请求到服务器。

**验收标准**:
- 当用户点击注册按钮时，系统应验证所有输入项
- 当存在输入验证错误时，系统应显示相应的错误提示
- 当所有输入验证通过时，系统应显示加载状态
- 当所有输入验证通过时，系统应发送注册请求到服务器
- 当注册请求发送中时，系统应禁用注册按钮防止重复提交

**前置条件**:
- 用户已填写注册表单

**后置条件**:
- 注册请求已发送或验证错误已显示

#### REQ-REG-007: 注册成功处理
**类型**: 功能需求
**优先级**: 高
**描述**: 当注册请求成功时，系统应完成用户注册并自动登录。

**验收标准**:
- 当服务器返回注册成功时，系统应显示"注册成功"提示
- 当服务器返回注册成功时，系统应保存用户信息和Token
- 当服务器返回注册成功时，系统应更新全局登录状态
- 当注册成功后，系统应在1秒后自动返回上一页面

**前置条件**:
- 注册请求已发送
- 服务器返回成功响应

**后置条件**:
- 用户已注册并登录
- 用户信息已保存

#### REQ-REG-008: 注册失败处理
**类型**: 功能需求
**优先级**: 高
**描述**: 当注册请求失败时，系统应显示相应的错误提示。

**验收标准**:
- 当服务器返回用户名已存在时，系统应提示"用户名已被注册"
- 当服务器返回参数错误时，系统应提示"注册信息有误，请检查"
- 当网络连接失败时，系统应提示"网络不可用，请检查网络连接"
- 当服务器内部错误时，系统应提示"服务器异常，请稍后再试"
- 当注册失败时，系统应隐藏加载状态并启用注册按钮

**前置条件**:
- 注册请求已发送
- 服务器返回失败响应或网络错误

**后置条件**:
- 错误提示已显示
- 用户可重新尝试注册

### 3.4 页面交互

#### REQ-REG-009: 密码显示切换
**类型**: 功能需求
**优先级**: 中
**描述**: 系统应允许用户切换密码的显示状态。

**验收标准**:
- 当用户点击密码输入框的显示/隐藏图标时，系统应切换密码的明文/密文显示状态
- 当用户点击确认密码输入框的显示/隐藏图标时，系统应切换确认密码的明文/密文显示状态

**前置条件**:
- 用户正在填写密码或确认密码

**后置条件**:
- 密码显示状态已切换

#### REQ-REG-010: 返回登录页面
**类型**: 功能需求
**优先级**: 中
**描述**: 系统应允许用户从注册页面返回登录页面。

**验收标准**:
- 当用户点击"已有账号？去登录"链接时，系统应导航到登录页面
- 当用户点击页面返回按钮时，系统应返回到登录页面

**前置条件**:
- 用户在注册页面

**后置条件**:
- 已返回登录页面

## 4. 非功能需求

### 4.1 性能需求
- 注册页面加载时间应不超过1秒
- 输入验证响应时间应不超过100毫秒
- 注册请求超时时间应设置为10秒

### 4.2 安全需求
- 密码在传输前应进行加密处理
- 用户Token应使用加密方式存储
- 注册请求应使用HTTPS协议传输
- 密码输入框默认应显示为密文状态

### 4.3 可用性需求
- 错误提示信息应清晰明确，便于用户理解
- 表单应支持键盘输入和触摸操作
- 输入框应提供适当的占位提示文本
- 加载状态应有明确的视觉反馈

### 4.4 兼容性需求
- 注册功能应兼容HarmonyOS手机和手表设备
- UI布局应适配不同屏幕尺寸
- 应遵循HarmonyOS设计规范

## 5. 约束条件

### 5.1 技术约束
- 使用ArkTS语言开发
- 遵循现有登录功能的架构模式
- 复用现有的HttpUtil、LoginStorageUtil等工具类
- 使用Preferences进行本地数据存储

### 5.2 业务约束
- 用户名长度限制为3-20位
- 密码长度限制为6-20位
- 用户名只能包含字母、数字和下划线
- 用户名在系统中必须唯一

## 6. 数据需求

### 6.1 输入数据
| 数据项 | 数据类型 | 必填 | 校验规则 | 说明 |
|-------|---------|-----|---------|-----|
| username | string | 是 | 3-20位字母数字下划线 | 用户名 |
| password | string | 是 | 6-20位字符 | 密码 |
| confirmPassword | string | 是 | 与password一致 | 确认密码 |

### 6.2 输出数据
| 数据项 | 数据类型 | 说明 |
|-------|---------|-----|
| userId | string | 用户唯一标识 |
| userName | string | 用户名 |
| avatarUrl | string | 用户头像URL（可选） |
| token | string | 认证Token |

## 7. 接口需求

### 7.1 用户界面
- 注册页面应与登录页面风格保持一致
- 使用应用Logo作为页面标识
- 输入框应使用统一的样式组件
- 按钮应使用应用主题色

### 7.2 系统接口
**注册接口**:
- 接口路径: `/api/user/register`
- 请求方法: POST
- 请求参数: `{ username: string, password: string }`
- 响应格式:
  ```json
  {
    "code": number,
    "message": string,
    "data": {
      "userId": string,
      "userName": string,
      "avatarUrl": string,
      "token": string
    }
  }
  ```

**响应码定义**:
- 0: 注册成功
- 1004: 用户名已存在
- 1003: 参数错误
- 5000: 服务器错误
